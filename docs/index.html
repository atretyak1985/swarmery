The craft of agents. A vendor-neutral, multi-project Claude Code agent framework, distributed as a plugin marketplace: one shared core, opt-in domain packs, and a thin per-project overlay. Publish an improvement once — every project adopts it with /plugin update.

§ 01

Why it exists

Copying an agent system between projects rots fast. agentry replaces fork-and-sync with a package manager.

The predecessor model kept one "upstream" agent repo and synced it into each project with token-rewrite scripts (project-a → project-b → …). In practice this produced:

  • Mis-substitution damage — real cases found during migration: kubectl rewritten into "gcloud run get pods", Keycloak renamed to "Auth.js v5", Drizzle/Flyway collapsed into "Prisma-to-Prisma", helm lint turned into npm run build .
  • Drift — every consumer slowly diverged; fixes had to be re-ported N times.
  • Identity bleed — one consumer's docs still described it as a "UAV/drone-swarm platform"… while being a flower-delivery marketplace.

agentry uses the native Claude Code plugin/marketplace mechanism instead: components are semver-versioned, namespaced (core:tech-lead), and updatable in place. Projects pin a known-good version and adopt on bump — controlled blast-radius, zero manual porting.

§ 02

Architecture

Two layers meet in every Claude Code session: the global layer (plugins served from this marketplace) and the local layer (the project's own .claude/). Claude Code merges them natively; on a name collision the project-local component wins.

▣ agentry (GitHub marketplace) .claude-plugin/marketplace.json — registers the four plugins below
│ enabled per project via settings.json │
core 1.0.041 agents · 23 skills · 14 commands · 13 hooks · CLI · templates. Always on.
uav-pack 1.0.0drones: MAVLink, telemetry, missions, embedded edge
iot-pack 1.0.0devices: BLE, telemetry ingest, health metrics
web-pack 1.0.0marketing: SEO, i18n, landing CRO, figma-style
▼ merged at session start ▼
<project>/.claude/ (local overlay — wins on collision) settings.json (which plugins + env) · project.json (flavor) · agents/ skills/ templates/ — only the project-unique few
▼ work artifacts go to ▼
agent-workspace/<project>/ (separate private repo) wiki/ · workspace/{working, archive, plans, specs, sessions*, logs*, metrics*} — *churn, gitignored
Framework ≠ workspace: the marketplace repo ships zero work-logs; plans/sessions/wiki live in a sibling private repo, namespaced per project.
Vendor-neutral core

No consumer project is privileged. Zero project tokens in plugins/** — enforced by a scan.

Runtime flavor

Agents, skills, and hooks read .claude/project.json at runtime for repos, apps, cloud, and domain nouns. Nothing is baked in.

Explicit semver

Each plugin carries a version. Consumers adopt on bump via /plugin update — a bad edit can't instantly break every project.

Opt-in domain packs

Domain vocabulary is legitimate inside its own pack, forbidden in core. Projects enable only what they need.

§ 04

How resolution works

  1. Session start. Claude Code reads the project's .claude/settings.json, fetches known marketplaces, installs/activates enabledPlugins (first time shows a trust prompt).
  2. Namespacing. Plugin components arrive namespaced: agent core:tech-lead, command /core:search, skill core:testing. No collisions between packs.
  3. Overlay merge. Anything in the project's own .claude/agents|skills|commands/ loads too. Same name as a plugin component → local wins. That's how a project intentionally overrides core behavior without forking the framework.
  4. Runtime flavor. Core components never hard-code project facts. They read .claude/project.json — repo list, main app, device repo, staging alias, cloud region, commit scopes, domain nouns — and the project's CLAUDE.md for everything else.
  5. Hooks. The core plugin wires its lifecycle hooks via hooks/hooks.json using ${CLAUDE_PLUGIN_ROOT} paths — the consumer's settings.json carries no hook wiring of its own.
  6. Workspace. Anything that writes work artifacts resolves $AGENT_PROJECT + $AGENT_WORKSPACE_ROOT (set in settings env) → agent-workspace/<project>/workspace/…, with a legacy .claude-workspace fallback.
Design rule

Flavor is configuration, not code. If you're about to write a project name into a framework file — stop, add a field to project.json instead.

§ 05

Quickstart — connect a new project

Four steps, ~10 minutes. Model files live in overlays/example/.

  1. Enable the marketplace + plugins

    Merge into <project>/.claude/settings.json:

    settings.json{
  "extraKnownMarketplaces": {
    "agentry": { "source": { "source": "github", "repo": "atretyak1985/agentry" } }
  },
  "enabledPlugins": {
    "core@agentry": true,
    "web-pack@agentry": true          // + iot-pack / uav-pack as needed
  },
  "env": {
    "AGENT_PROJECT": "my-project",      // workspace + overlay key (kebab-case)
    "AGENT_WORKSPACE_ROOT": "/path/to/agent-workspace"
  }
}

  2. Describe the project — project.json

    Copy overlays/example/project.json to <project>/.claude/project.json and fill it (full field reference in § 06):

    .claude/project.json{
  "name": "my-project",
  "codePath": "/path/to/my-project",
  "mainApp": "web-app",
  "repos": ["apps/web-app", "apps/docs", "infrastructure"],
  "cloud": { "provider": "gcp", "region": "…", "envAlias": "staging" },
  "commitScopes": ["web-app", "docs", "infra"],
  "enabledPacks": ["web-pack"]
}

  3. Add the workspace namespace

    shellmkdir -p agent-workspace/my-project/{wiki,workspace/{working,archive,plans,specs,sessions,logs,metrics}}

    (Churn dirs — sessions/ logs/ metrics/ — are gitignored by the workspace repo.)

  4. Start a fresh session & verify

    • Accept the marketplace trust prompt → plugins install.
    • @core:tech-lead resolves; /core:search works.
    • The session banner shows your displayName; hooks fire once.
    • agent-work.sh init "hello" research lands under agent-workspace/my-project/workspace/working/…
Cutover caution — migrating an EXISTING agent setup

If the project previously ran a file-based copy of this system with hooks wired in its settings.json, remove that legacy hook wiring in the same change that enables the plugins — otherwise every hook fires twice. Do the switch from a terminal, then start a fresh session.

§ 07

Workspace & CLI

Work artifacts never live in the framework repo. The CLI (plugins/core/bin/agent-work.sh) resolves $AGENT_WORKSPACE_ROOT/$AGENT_PROJECT/workspace and manages the task lifecycle:

agent-work.shsetup                       # create workspace directories
init <name> [type]          # new task → working/YYYY/MM/DD/<slug>  (audit|feature|refactor|research|incident|infra)
phase <id|--latest> <NN-x>  # scaffold phases/NN-x.md from template
complete <id|--latest>      # README→done + SUMMARY skeleton + move to archive/
index                       # regenerate INDEX.md (also runs from the session-summary hook)
list [active|done] · search <q> · view <id> · metrics · cleanup [days]
  • Task id = yyyy-mm-dd-slug; the folder is working/YYYY/MM/DD/<slug>/ with a README card, plan/ specs/ reports/ logs/.
  • Durable vs churn: wiki/ plans/ specs/ working/ archive/ are committed; sessions/ logs/ metrics/ are gitignored.
  • Wiki (agent-workspace/<project>/wiki/ + shared _shared/wiki/) holds durable knowledge notes — flat Markdown with frontmatter and [[wikilinks]]; open it as an Obsidian vault for a graph view.
  • The core session-summary hook mirrors session stats into metrics/, links sessions to touched tasks, and regenerates the INDEX — automatically, per project.
§ 08

Extending — where new things go

The thing is…It goes to…
useful to any projectplugins/core — bump semver, everyone adopts via /plugin update
useful to ≥2 projects of one domainthe domain pack (uav / iot / web)
unique to one projectthe project's own .claude/ — versioned with the product code
configuration, not logic.claude/project.json
The graduation rule — flow goes UP only

New things are born project-local. When a second project needs the same thing → promote to a pack. When everyone needs it → promote to core (de-flavor first, bump semver, delete the donor's local copy). Never copy downward — that recreates the fork-rot this framework exists to kill.

  • Overrides: ship a local component with the same name to replace a core one in that project only.
  • Templates: agents check <project>/.claude/templates/ first, then ${CLAUDE_PLUGIN_ROOT}/templates/.
  • Health check: a consumer's .claude/ should stay thin. If it accumulates generic content, something wants promoting.
§ 09

Vendor neutrality

The invariant that makes sharing possible: plugins/** contains zero project tokens.

  • Brand tokens (company/product names, internal repo names, env aliases, regions) — forbidden everywhere in plugins.
  • Domain vocabulary (drones for uav-pack, wearables for iot-pack) — legitimate inside its own pack, forbidden in core.
  • Prose uses neutral placeholders (apps/<mainApp>, <device>, <envAlias>) and neutral example domains (orders/line-items, pipelines/job-runs).
checker# your private token regexes (gitignored) drive the scan:
echo 'mycompany|my-app|my-env' > .flavor-tokens
echo 'my-domain-noun'        > .flavor-tokens-domain
bash scripts/scan-flavor.sh     # target: 0 occurrences — run as a CI ratchet

CI (.github/workflows/ci.yml) validates all plugin JSON, bash -n's every script, checks agent frontmatter, and runs the neutrality scan (patterns from repo vars). A promptfoo eval suite (evals/) guards agent behavior contracts — routing, commit-scope derivation, guardrail verdicts.

§ 10

Versioning & updates

  1. Change lands in plugins/core → bump version in its plugin.json (semver) → push.
  2. Each consumer runs /plugin update when ready — version pinning means nobody is force-fed HEAD.
  3. If a release misbehaves: consumers simply don't update (or re-pin); fix forward in the marketplace.
The payoff test

Bump core minor → push → /plugin update in every consumer → confirm one improvement landed in all projects with zero per-project file copying. This is the entire reason agentry exists; verify it explicitly once ≥2 consumers are live.

Version resolution order (Claude Code): plugin.json version → marketplace entry version → git SHA. If you push changes without bumping the version, consumers keep the cached copy — bump or it didn't happen.
